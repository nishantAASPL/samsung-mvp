# Samsung Predictive Inventory Platform — Build Guide

> **MVP Status:** This is a prototype/demo system. The "AI" engine is a deterministic
> MTBF formula — no machine-learning model is running. All inventory predictions
> are calculated from fleet demographics (size, age) vs engineering failure curves.

---

## 1. Project Overview

The platform demonstrates the business impact of replacing **static reorder rules**
with **MTBF-driven predictive thresholds** across Samsung's service-center network.

| Dimension       | AS-IS (Static Rules)    | TO-BE (MTBF Engine)          |
|-----------------|-------------------------|------------------------------|
| FTFR            | 52%                     | 78% (+26 pp projected)       |
| Stockout Events | 482 / year              | 0 (pre-empted)               |
| Dead Stock      | Rs. 14.5L locked in system | Rs. 4.2L (-71%)           |
| Avg Wait Time   | 185 mins                | 12 mins                      |
| Reschedules     | 1,240 / year            | 0 (intercepted)              |

**Scope:** 5 Samsung service-center partners across India, 5 device models
(Galaxy S23 Ultra, S23, A54, A34, M34), 4 part categories.

## 5. Dataset Schemas

> These schemas define the **data format the customer (Samsung OEM / service-center
> partners) must supply** for the platform to run.  The CSVs generated by
> `generate_data.py` are synthetic examples of correctly formatted input.

---

### 5.1 `partners_master.csv`

One row per service-center partner. Supplied by the OEM during onboarding.

| Column                  | Type    | Example           | Description                               |
|-------------------------|---------|-------------------|-------------------------------------------|
| `partner_id`            | string  | `P1`              | Unique partner code (system-assigned)     |
| `partner_name`          | string  | `SwiftFix Delhi`  | Legal trading name of the service center  |
| `city`                  | string  | `New Delhi`       | City of operation                         |
| `region`                | string  | `North`           | Broad region (North / West / South / East)|
| `sla_tier`              | string  | `Gold`            | SLA tier: `Platinum`, `Gold`, or `Silver` |
| `capacity_jobs_per_day` | integer | `35`              | Maximum repair jobs the center can handle |
| `contact_email`         | string  | `ops@swiftfix.in` | Operational contact for automated alerts  |

**SLA tier rules (used by allocation engine):**

| Tier     | Priority | Safety Stock Multiplier |
|----------|----------|-------------------------|
| Platinum | 1 (High) | 1.5x                    |
| Gold     | 2        | 1.3x                    |
| Silver   | 3 (Low)  | 1.1x                    |

### 5.3 `parts_catalog.csv`

Spare parts master with engineering failure data. Supplied by Samsung parts division.

| Column                    | Type    | Example                 | Description                                                  |
|---------------------------|---------|-------------------------|--------------------------------------------------------------|
| `part_code`               | string  | `PT01`                  | Unique part identifier                                       |
| `part_name`               | string  | `OLED Display Assembly` | Human-readable part name                                     |
| `part_category`           | string  | `Display`               | `Display`, `Battery`, `Charging`, `Housing`                  |
| `compatible_models`       | string  | `SM-S918B`              | Comma-separated model codes this part fits                   |
| `mtbf_months`             | integer | `28`                    | **Mean Time Between Failures in months** — core MTBF engine input |
| `unit_cost_inr`           | integer | `12000`                 | Replacement part cost in INR                                 |
| `supplier_lead_time_days` | integer | `21`                    | Days from PO to physical delivery at warehouse               |
| `is_critical`             | boolean | `true`                  | If `true`, allocator prioritises this part in scarce-stock scenarios |

> **MTBF is the most important field.** It drives every dynamic threshold and
> failure-wave prediction in the system. It must come from Samsung's engineering
> reliability database, not from field estimates.

---

### 5.4 `fleet_registry.csv`

Active device count per (partner, model) — a periodic demographic snapshot.
Supplied by the OEM from device activation data. Updated monthly.

| Column                  | Type    | Example      | Description                                                    |
|-------------------------|---------|--------------|----------------------------------------------------------------|
| `snapshot_date`         | date    | `2025-06-30` | Date this snapshot was taken (`YYYY-MM-DD`)                    |
| `partner_id`            | string  | `P3`         | Matches `partners_master.partner_id`                           |
| `model_code`            | string  | `SM-A546B`   | Matches `device_models.model_code`                             |
| `model_name`            | string  | `Galaxy A54` | Denormalised for readability                                   |
| `active_device_count`   | integer | `1120`       | Number of this model registered in the partner's territory     |
| `fleet_avg_age_months`  | integer | `28`         | **Average age of the fleet in months** — core MTBF engine input|
| `fleet_activation_from` | date    | `2023-02-28` | Earliest activation date in the cohort                         |
| `fleet_activation_to`   | date    | `2025-06-30` | Latest activation date in the cohort                           |

> **`fleet_avg_age_months` is the second most important field.**
> Pairing this with `mtbf_months` from the parts catalog tells the engine exactly
> where the fleet sits on the failure curve.
>
> Example: age 28m / MTBF 14m = entering 2nd failure cycle -> trigger pre-order.

## 7. MTBF Engine Logic

The "AI" is a two-variable deterministic formula. No model is trained or inferred.

```
Dynamic Threshold = max(2, round(monthly_demand x coverage_months x sla_multiplier))

Where:
  monthly_demand  = (fleet_size / mtbf_months) x intensity
  intensity       = 0.05 + 0.95 x (cycle_position ^ 1.5)   <- accelerates near wave
  cycle_position  = (fleet_age_months % mtbf_months) / mtbf_months   [0 to 1]
  coverage_months = ceil(supplier_lead_time_days / 30) + 1
  sla_multiplier  = Platinum: 1.5  |  Gold: 1.3  |  Silver: 1.1
```

**Worked example — P3 A54 Battery:**

```
fleet_size      = 1,120 devices
fleet_age       = 28 months
mtbf_months     = 14 months
cycle_position  = (28 % 14) / 14 = 0.0   <- exactly entering 2nd cycle
intensity       ~ 0.05                    <- bottom of new cycle, about to spike
monthly_demand  ~ 4 to 5 failures/month
coverage_months = ceil(10/30) + 1 = 2
sla_multiplier  = 1.5 (Platinum)
-> Dynamic Threshold = max(2, round(5 x 2 x 1.5)) = 15 (calibrate with real data -> 42)
```

**Business rules layered on top:**

| Rule                   | Logic                                                              |
|------------------------|--------------------------------------------------------------------|
| Dead Stock Detection   | If `stock > threshold x 10`, flag for inter-partner transfer       |
| Allocation Priority    | Platinum > Gold > Silver when warehouse has insufficient stock      |
| Wave Pre-order Lead    | Orders triggered 30 days before `months_to_next_wave <= 3`         |
| Inter-Partner Transfer | Excess stock at Partner A auto-routed to Partner B at Rs. 0 cost   |

## 9. Customer Data Delivery Checklist

When requesting data from Samsung OEM / service-center partners, ask for:

- [ ] **`partners_master.csv`** — from Samsung partner-management system
- [ ] **`device_models.csv`** — from Samsung product database
- [ ] **`parts_catalog.csv`** — from Samsung engineering / parts division
      *(MTBF values must come from reliability engineering, not estimated)*
- [ ] **`fleet_registry.csv`** — from Samsung device activation / telemetry system
      *(must include `fleet_avg_age_months` per partner per model)*
- [ ] **`repair_history.csv`** — from partner JMS/CRM, minimum 12 months, ideally 18+
- [ ] **`inventory_snapshots.csv`** — from partner IMS, minimum 6 months of history
- [ ] **`reservations_pipeline.csv`** — from OEM booking portal, 3-6 months forward

**Minimum viable data for the engine to run:** `parts_catalog` + `fleet_registry`

All other datasets improve accuracy and enable the analytics dashboards.
